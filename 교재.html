Phase 1 — JS 핵심
Chapter 01

Java vs JS
변수 · 함수 · 타입

Java를 알면 JS의 절반은 이미 압니다. 나머지 절반, 다른 점만 빠르게 정리합니다.

1. 변수 선언

Java는 타입을 명시하고 선언합니다. JS는 타입 없이 const / let으로 선언합니다.

☕ Java
String name = "자금일보";
int amount = 50000;
boolean isNew = true;

// final = 상수 (재할당 불가)
final String APP = "jafm";
🟨 JavaScript
let name = "자금일보";
let amount = 50000;
let isNew = true;

// const = 상수 (재할당 불가)
const APP = "jafm";
💡

const vs let 언제 쓰나?

나중에 값이 바뀔 변수 → let / 바뀌지 않을 변수 → const
실제 소스에서 transactions는 배열 자체를 교체하므로 let, isElectron은 한 번 정해지면 바뀌지 않으므로 const를 사용했습니다.

실제 소스 — state.jsJS
// 거래내역은 계속 추가/삭제되므로 let
let transactions = [];

// Electron 여부는 앱 실행 중 변하지 않으므로 const
const isElectron = typeof window.electronAPI !== 'undefined';

2. 함수 선언 — 3가지 방식

Java는 메서드 하나뿐이지만 JS는 함수를 선언하는 방식이 3가지입니다.

함수 선언 3가지JS
// ① 함수 선언문 (Java 메서드와 가장 비슷, 호이스팅 됨)
function calcBalance() {
  return 50000;
}

// ② 함수 표현식 (변수에 함수를 담음)
const calcBalance = function() {
  return 50000;
};

// ③ 화살표 함수 (가장 많이 씀, this 바인딩 없음)
const calcBalance = () => 50000;

// 화살표 함수 — 한 줄이면 return 생략 가능
const double = (n) => n * 2;
const add    = (a, b) => a + b;
실제 소스 — accounts.jsJS
// ① 선언문: 이름이 있고 독립적인 로직
function calcBalance() {
  const bal = {};
  ACCOUNTS.forEach(a => bal[a.key] = INIT_BALANCE[a.key] || 0);
  return bal;
}

// ③ 화살표 함수: forEach, map 등 콜백으로 자주 사용
ACCOUNTS.forEach(a => {
  // a는 각 계좌 객체 — Java의 for-each와 동일
  total += bal[a.key] || 0;
});

Phase 1.5 — Java to JavaScript
특별 챕터

람다(Lambda) 완전 정복

Java 람다부터 JavaScript 화살표 함수까지, "메서드를 변수처럼 다루는" 개념을 실무 예제로 완벽하게 이해합니다.

1. 람다가 뭔데? (기본 개념)

람다는 "메서드를 변수처럼 다루는 문법"입니다. 예전에는 간단한 로직 하나 쓰려고 클래스를 만들어야 했는데, 람다는 그걸 한 줄로 줄여줍니다.

예전 방식 vs 람다

✗ 예전 Java (클래스 필요)
// 리스트 출력하려고 클래스 만들기
for (Transaction tx : txList) {
    System.out.println(tx);
}
✓ 람다 (한 줄로 됨)
// 람다로 간결하게
txList.forEach(tx -> System.out.println(tx));
💡

핵심

tx -> 이 부분이 람다식. "각 tx마다 뭘 할지"를 화살표 오른쪽에 써주면 됩니다.

2. 람다 기본 문법

Java
// 기본 형태
(파라미터) -> { 실행할 코드 }

// 예시 1: 리스트 출력
List<Integer> numbers = Arrays.asList(1, 2, 3, 4, 5);
numbers.forEach(num -> System.out.println(num));

// 예시 2: 여러 줄일 때는 중괄호 사용
numbers.forEach(num -> {
    if (num % 2 == 0) {
        System.out.println("짝수: " + num);
    }
});
⚠️

변수명은 내가 정해!

num, item, tx, x 뭐든 상관없어요. for (int num : numbers)에서 num 정하는 것과 똑같습니다.

3. 람다 응용: forEach 말고 더 뭐가 있어?

① filter - 조건 필터링

③ sort - 정렬

④ 실무 조합 예시

Java - 실무 스타일

4. 왜 람다가 유용해? (메서드를 파라미터로 넘기기)

람다의 진짜 목적은 "처리 로직을 나중에 결정"하기 위해서입니다.

예전 문제점

Java
// 간단한 로직 하나 쓰려고 클래스 3개 만들기
class PrintProcessor implements Processor {
    public void process(Transaction tx) {
        System.out.println(tx);
    }
}
class LogProcessor implements Processor { ... }
class EmailProcessor implements Processor { ... }

processAll(new PrintProcessor());
processAll(new LogProcessor());
processAll(new EmailProcessor());

람다로 해결

Java
// 클래스 없이 "로직만" 전달
processAll(tx -> System.out.println(tx));
processAll(tx -> log.save(tx.getId()));
processAll(tx -> email.send(tx.getAmount()));

// processAll 메서드 정의
public void processAll(Consumer<Transaction> processor) {
    for (Transaction tx : txList) {
        processor.accept(tx);  // 받은 로직 실행
    }
}
✅

"메서드를 파라미터로 넘긴다" = 처리 방식을 나중에 결정

processAll은 "리스트 순회"만 하고, "어떻게 처리할지"는 호출하는 쪽이 람다로 전달합니다.
이게 바로 콜백(Callback) 패턴이에요.

5. 람다 ≠ 스트림 (완전히 다른 개념!)

⚠️

헷갈리기 쉬운 부분

람다 = 화살표 문법 (x -> x * 2)
스트림 = 데이터 처리 파이프라인 (.stream().filter().map())
둘은 자주 같이 쓰이지만 별개의 개념입니다!

람다만 (스트림 없이)
// forEach에 람다만 사용
list.forEach(item -> System.out.println(item));

// 버튼 클릭 (콜백)
button.onClick(() -> System.out.println("클릭"));

// 스레드 실행
executor.submit(() -> doWork());
스트림 + 람다
// 스트림에서 람다 사용
list.stream()
    .filter(x -> x > 10)  // 람다
    .map(x -> x * 2)      // 람다
    .collect(Collectors.toList());

6. 스트림이 for문보다 확실히 좋은 경우

① 병렬 처리 (멀티코어 활용)

Java
// for문 - 순차 처리 (느림)
for (Transaction tx : txList) {  // 10초
    heavyCalculation(tx);  // CPU 집약적
}

// 스트림 - 병렬 처리 (빠름)
txList.parallelStream()  // 3초로 단축!
    .forEach(tx -> heavyCalculation(tx));

② 데이터 그룹화/집계

Java
// for문 - 복잡함
Map<String, List<Transaction>> grouped = new HashMap<>();
for (Transaction tx : txList) {
    grouped.computeIfAbsent(tx.getStatus(), k -> new ArrayList<>()).add(tx);
}

// 스트림 - 깔끔
Map<String, List<Transaction>> grouped = txList.stream()
    .collect(Collectors.groupingBy(Transaction::getStatus));

③ 함수형 체이닝 (가독성)

Java
// for문 - 단계별로 나눠야 함
List<Transaction> filtered = new ArrayList<>();
for (Transaction tx : txList) {
    if (tx.getAmount() > 50000) filtered.add(tx);
}
List<String> ids = new ArrayList<>();
for (Transaction tx : filtered) { ids.add(tx.getId()); }

// 스트림 - 하나의 흐름
List<String> ids = txList.stream()
    .filter(tx -> tx.getAmount() > 50000)
    .map(Transaction::getId)
    .sorted()
    .collect(Collectors.toList());

7. for문이 스트림보다 좋은 경우

성능 비교 (실측!)

Java - 성능 테스트
// for문 방식
for (Account acc : accountList) {  // 1번 순회

성능 진실

하지만 차이는 미미하고 (밀리초 단위), 가독성이 훨씬 중요합니다.
단, API/DB 작업처럼 무거운 작업에서는 for문이 디버깅하기 더 쉬워요.

Java: 멀티 스레드 → 별도 스레드에서 파일 I/O 처리

1. 화살표 함수 (Arrow Function)

React 코드에서 function 키워드 대신 =>를 매우 많이 씁니다.
기능은 같고, 쓰는 방식만 다릅니다.

일반 함수 vs 화살표 함수
// 일반 함수
function add(a, b) {
  return a + b;
}

// 화살표 함수 — 완전한 형태
const add = (a, b) => {
  return a + b;
};

// 화살표 함수 — 한 줄이면 return과 {} 생략 가능
const add = (a, b) => a + b;

// 파라미터가 하나면 () 도 생략 가능
const double = n => n * 2;

React 코드에서 보이는 패턴

화살표 함수가 쓰이는 곳
// 버튼 클릭 이벤트
<button onClick={() => setCount(count + 1)}>+</button>
//              ↑ 파라미터 없으면 () 필수

// 배열 map
transactions.map(tx => <TxRow key={tx.id} tx={tx} />)
//           ↑ tx 하나씩 받아서 JSX 반환

// 이벤트 핸들러에서 인자 전달할 때
<button onClick={() => onDelete(tx.id)}>삭제</button>
// () => 로 한 번 감싸야 클릭할 때 실행됨
// onClick={onDelete(tx.id)} 라고 쓰면 즉시 실행되어 버림 ❌
⚠️

onClick 함정 — 가장 많이 하는 실수

onClick={onDelete(tx.id)} → 렌더링될 때 즉시 실행 (버튼 클릭 안 해도!) ❌
onClick={() => onDelete(tx.id)} → 클릭할 때만 실행 ✅

1. useState 원리 — 왜 직접 변수를 못 쓰나?

React 컴포넌트는 state가 바뀔 때마다 함수 전체가 다시 실행됩니다.
그러므로 let count = 0처럼 로컬 변수를 쓰면 매번 0으로 초기화됩니다.

useState 내부 동작 (개념)
// 이런 식으로 동작한다고 이해하면 됨
let _state = [];          // React가 외부에 보관

function useState(initial) {
  if (_state === undefined) _state = initial;

  function setState(newValue) {
    _state = newValue;
    rerender();           // 컴포넌트 다시 실행!
  }
  return [_state, setState];
}

// 컴포넌트 함수는 매 렌더마다 실행되지만,
// _state는 React가 외부에 보관하므로 값이 유지됨

state 업데이트 패턴

올바른 state 업데이트
const [transactions, setTransactions] = useState([]);

// ❌ 잘못된 방법 — 직접 수정은 React가 변화를 감지 못함
transactions.push(newTx);
setTransactions(transactions);  // 같은 배열 참조라 변화 없음

// ✅ 올바른 방법 — 새 배열을 만들어서 전달
setTransactions(prev => [...prev, newTx]);

// ✅ 삭제도 마찬가지
setTransactions(prev => prev.filter(t => t.id !== id));
⚠️

배열/객체는 반드시 새로 만들어서 set

React는 참조(주소)를 비교합니다. push로 기존 배열을 수정하면 같은 주소이므로 바뀐 것으로 인식하지 않습니다.
스프레드([...prev, newTx])나 map/filter로 새 배열을 만들어야 합니다.

2. useEffect — 사이드이펙트를 분리하는 Hook

사이드이펙트(Side Effect)란 렌더링과 직접 관계없는 작업입니다.
예: API 호출, localStorage 저장, 이벤트 리스너 등록, 타이머 설정

🟨 Vanilla JS (app.js)
// init 함수에서 초기화
(async function init() {
  await loadInitBalance();
  await loadData();
  renderAll();
})()
⚛️ React — useEffect
function App() {
  const [txs, setTxs] = useState([]);

  useEffect(() => {
    // 컴포넌트가 처음 화면에 나타날 때 실행
    loadData().then(data => setTxs(data));
  }, []);  // ← [] : 최초 1회만
}

useEffect의 세 가지 패턴

dependency array 패턴
// 1. 매 렌더마다 실행 (dependency 없음)
useEffect(() => {
  saveData(transactions);
});

// 2. 최초 1회만 실행 (빈 배열)
useEffect(() => {
  loadData();
}, []);

// 3. 특정 값이 바뀔 때만 실행
useEffect(() => {
  saveData(transactions);
}, [transactions]);  // transactions 바뀔 때마다

cleanup — 정리 함수

이벤트 리스너 등록 / 해제
useEffect(() => {
  // 등록
  const handleResize = () => console.log('resize');
  window.addEventListener('resize', handleResize);

  // cleanup: 컴포넌트가 사라질 때 자동 실행
  return () => {
    window.removeEventListener('resize', handleResize);
  };
}, []);

3. React 생명주기 vs useEffect

클래스 컴포넌트 시대의 생명주기 메서드는 함수 컴포넌트에서 useEffect 하나로 통합됩니다.

클래스 컴포넌트함수 컴포넌트 (useEffect)시점
componentDidMountuseEffect(() => {}, [])최초 마운트 후
componentDidUpdateuseEffect(() => {}, [deps])deps 변경 후
componentWillUnmountuseEffect(() => { return () => {} }, [])언마운트 전
✅

요즘 React는 함수형 컴포넌트가 표준

2019년 Hook 도입 이후 클래스 컴포넌트는 더 이상 권장하지 않습니다.
새 프로젝트라면 처음부터 함수형 + Hook으로 작성하세요.

GitHub Actions — 자동 빌드 · 배포

코드를 push하면 GitHub이 자동으로 Windows 설치 파일을 만들어주는 CI/CD 파이프라인을 구축합니다.

1. CI/CD란?

용어뜻자금일보에서
CI (Continuous Integration)코드 변경 시 자동 검증push하면 빌드 테스트 자동 실행
CD (Continuous Delivery)자동으로 배포 가능한 상태 유지태그 생성 시 .exe 자동 생성
💡

GitHub Actions는 무료

Public 저장소는 무제한 무료, Private 저장소는 월 2,000분 무료입니다.
Electron 빌드는 보통 3~5분 소요됩니다.

2. 워크플로우 파일 작성

GitHub Actions는 .github/workflows/ 폴더에 YAML 파일로 정의합니다.

.github/workflows/build.yml
name: Build Electron App

on:
  push:
    tags:
      - 'v*'           # v1.0.0 같은 태그 push할 때만 실행

jobs:
  build-windows:
    runs-on: windows-latest   # Windows 환경에서 실행

    steps:
      # 1. 코드 체크아웃
      - uses: actions/checkout@v4

      # 2. Node.js 설치
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      # 3. 의존성 설치
      - name: Install dependencies
        run: npm ci

      # 4. 빌드
      - name: Build Windows installer
        run: npm run build:win

      # 5. 결과물을 GitHub Releases에 업로드
      - name: Upload to GitHub Releases
        uses: softprops/action-gh-release@v2
        with:
          files: dist/*.exe
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
